# Trust-Lean Benchmarks (v1.1.0)

## Criteria

Rubric generated by `/benchmark-qa --strict` during planning for the ExpandedSigma → Stmt bridge.
This section is written ONCE and not modified during execution.

### N6.1: FUNDACIONAL (Bridge Types + Semantics)

**Objective**: Ensure the semantic foundation is sound, deterministic, and correctly models the source language.

#### 1. Checks Mecánicos Obligatorios
- `lake build` with zero errors and zero warnings
- `nolint` passes on all files
- 0 `sorry`, 0 `axiom`, 0 `partial`/`unsafe` in definitions
- All wrapper types compile independently

#### 2. Stress Testing
- **VarName Mapping**: Test with >1000 unique variable names, including names that are substrings of others (e.g., "x", "x1", "x10", "x100")
- **Denotational Semantics**: Evaluate expressions that result in boundary values (`Int.min`, `Int.max`, `0`, `1`, `-1`)
- **Division by Zero**: Test `evalScalarExpr` on division by a variable that evaluates to zero (must return `none` or handle gracefully)

#### 3. Edge Cases
- **Wrapper Types**: Test with empty or singleton `amo-lean` structures
- **VarName Mapping**: Test with an empty set of input variables
- **Semantics**: `evalScalarExpr` on a literal constant and on a single variable
- **Degenerate Inputs**: Empty arrays, zero-dimensional tensors

#### 4. Robustness
- **Compositionality**: The definition of `evalScalarExpr` for an operator (e.g., `add`) must only depend on the evaluation of its sub-expressions, not their structure
- **Extensibility**: Adding a new `amo-lean` type should not require changes to existing wrapper types
- **Environment Independence**: Evaluation must not depend on global state or mutable references

#### 5. Proof Quality
- **Computability**: All semantic functions (`eval...`) must be provably terminating and computable (no non-constructive axioms)
- **`@[simp]` Hygiene**: `simp` lemmas derived from definitions should be minimal and non-looping. Verify by running `simp` on a simple goal and observing finite, small number of steps
- **No `native_decide`**: Avoid proof by evaluation on concrete examples
- **Structured Proofs**: Follow data type structure (e.g., `induction on ScalarExpr`)

#### 6. Architectural Coherence
- **Isomorphism**: For each wrapper type `W` and original type `A`, a formal proof of isomorphism (`W ≃ A`) or clear embedding/projection pair (`f: A → W`, `g: W → Option A`) must exist
- **Determinism**: All `eval` functions must be proven deterministic: `eval env expr = y₁ → eval env expr = y₂ → y₁ = y₂`
- **Type Safety**: Bridge predicate must ensure type compatibility between amo-lean and Trust-Lean types

### N6.2: CRÍTICO GATE (Expression Translation + Proofs)

**Objective**: Guarantee that expression-level translation is correct and that the variable mapping is sound.

#### 1. Checks Mecánicos Obligatorios
- `lake build` with zero errors and zero warnings
- `nolint` passes
- 0 `sorry`, 0 `axiom`
- All correctness proofs complete (6 cases for `scalarExprToLLExpr`, 5 cases for `idxExprToLLExpr`)

#### 2. Stress Testing
- **scalarExprToLLExpr**: Translate deeply nested expressions (`a+(b+(c+...))`) of depth > 20
- **Mixed Operators**: Translate expressions mixing all 6 operator types in a single expression
- **idxExprToLLExpr**: Translate index expressions that are themselves complex scalar expressions
- **Large Constants**: Test with constants near `Int.min` and `Int.max`

#### 3. Edge Cases
- Translate a single constant literal
- Translate a single variable
- Translate an index of `0` and an index involving a variable
- Translate identity operations (e.g., `x + 0`, `x * 1`)

#### 4. Robustness
- **Translation Modularity**: The translation of a sub-expression must be independent of its parent expression's context
- **Injectivity Proof**: The proof that `VarName` mapping is injective must be general and not rely on a concrete list of variables
